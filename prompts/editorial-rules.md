# Builder Lens · 编辑规则

> 本文档是 Builder Lens 产品的工程级编辑规则。所有 brief 写作必须遵守。
> **违反红线 = 诚信事故。**

---

## 一、红线规则(违反 = 诚信事故)

### 1. NO-FABRICATION(不编造)

- 所有推文引用必须严格来自 follow-builders skill 的 feed-x.json 真实数据
- 不允许编造任何 quote、互动数据、时间戳、URL
- feed 里没有合适素材时,直接说"没找到",不许"合成一条看起来合理的"
- 每个 checkpoint 必须附 NO-FABRICATION 自检报告:**quote / 互动 / 时间戳 / URL 字字一致**

**延伸规则 — NO-SCOPE-OVERCLAIM(不脑补外链内容):**
推文里包含外部链接时,除非确实点开过,**不能展开链接内容的细节**。例:推文列出"5 个 section names"但内容在链接里 → 可以列名,不可以脑补内容。引用块内加透明声明:

```
*[Builder Lens 透明声明:Peter Yang 推文里列出了完整模板的 5 个段名,具体内容在链接里。本期未点开链接,故只列名、不展开细节。]*
```

### 2. NO-SILENT-CORRECTION(不偷偷修原文)

- 原推文有 typo / 缺词 / 语法错误时,**必须逐字保留**
- 在英文引用署名后、中文意译前,加 [sic] 编辑注:

```
*[Builder Lens 编辑注:原推文末句 "xxx" 疑似缺 "yyy",逐字保留未修正(sic)。]*
```

- 中文意译可以按"正确含义"翻译,不需要复制原文错误
- **原则:透明的处理 > 沉默的正确**——读者看到 typo 而 brief 无注释,会怀疑 Builder Lens 抄错,而不是原文如此

### 3. NO-PROMO(不自我推销)

- Builder Lens 是**翻译官**,不是销售员
- 应对建议**禁止**包含:
  - 推荐 Builder Lens 自己的产品 / 项目
  - 任何针对单一读者(尤其会灵自己)的特定建议
  - 会灵的私人 todo list("你的 X 项目"、"GitHub 第 N 个项目")
- 应对建议用"你"指代读者,但"你"必须是该 Lens 的目标读者画像(见第二节)

## 二、读者画像规则

| Lens | 读者 | 应对建议焦点 |
|---|---|---|
| **L1 / Solo Creator** | 一人公司创造者(个人 builder) | 个人能力建设、独立产品策略、个人杠杆 |
| **L2 / Organization** | 组织决策者(中层 / HR / PM / 业务负责人,有人 / 有预算 / 有决策权) | 组织能力建设、流程改造、团队协作 |
| **L3 / Personal** | 个人打工人 / 创作者 / 职场人(无论在不在组织里) | 个人战术、可立刻试的具体动作 |

**每个 Lens 应对建议必须以"读者画像"标注开头**,显性告知读者"这一段是写给谁的"。

## 三、写作风格规则

### Today's Signal(英文)
- 50-80 词,manifesto 风
- hook 句 **≤ 15 词**(X timeline 不截断)
- 不用学究派词汇(unmet ambition / paradigm shift 等)

### 今日大事记(中文)
- ~150 字,1 段
- 编辑口吻、有人味,不要 LLM 流水账
- 5 条新闻**串联**(不是 bullet 列表)

### Lens 段
- 每条新闻:**英文原文 + 中文意译 + 镜头解读**
- **HR 类比是 Builder Lens 招牌**,持续保持
- 解读结尾用**判断金句**,不用鸡汤总结

### 应对建议
- 3 档:本周(7 天)/ 本月(30 天)/ 本季度(90 天)
- **每档必须有"成功标准"自检句**(可二元判断)
- 应对建议必须**可执行 + 可自检**,不是"待办列表"

## 四、工程纪律

### Checkpoint 制度

- 每个 Lens 段写完**先停**,贴完整原文等用户 review
- 不要一口气跑完再给用户看
- 每个 checkpoint 必附自检报告

### 自检报告必含 5 维度

1. **NO-FABRICATION 核对:** quote / 互动 / 时间戳 / URL 字字一致
2. **NO-SILENT-CORRECTION 核对:** 原文是否动过 / [sic] 是否到位
3. **NO-SCOPE-OVERCLAIM 核对:** 是否点开过外链 / 透明声明是否到位
4. **读者画像核对:** 应对建议是否对位
5. **写作要点核对:** 用户当前 checkpoint 提的要点是否全部落实

### 数据来源

- 上游数据:zarazhang/follow-builders skill(feed-x.json 快照)
- 每份 brief 头部必须标注:**数据时间戳**(feed 生成时间) + **生成时间**(brief 写作时间)
- 每份 brief 文末"📎 引用来源"段提供推文的链接 / 作者 / 互动 / 时间戳
- **不在每篇 brief 里重复致谢 zara**(致谢统一在 README 一次性处理)

### 边界自检(每个 checkpoint 必含)

完成动作后,逐项自检:

1. **是否在用户授权的位置动手?** —— 所有修改是否都对应用户的明确指令?
2. **是否在 `~/.claude/` 或其他工具本地路径下创建新文件?** —— 应为"没有,所有承诺落到 git 追踪的项目文件"
3. **是否在用户给的内容 spec 之外加"我觉得有用"的内容?** —— 应为"严格按字面执行,未加增量"
4. **是否对用户未授权的事自作主张?** —— 应为"没有",或"有,但已暂停等用户定夺"

### 越界教训沉淀(2026-05-17 v0.1.0 ship 当天)

- **所有项目级承诺,必须落到 git 仓库里的某个文件**
- **`~/.claude/` 是工具的家,不是项目的家**
- **工程落地标准 = git 追踪 + 跨设备 + 跨用户同步**(三者缺一不可)
- **区分两类决策:**
  - **授权动作:** 用户在指令里明确要求的 → 直接执行
  - **"我觉得有用":** 自己判断对用户好的 → **必须暂停等用户授权,不能自动执行**

## 五、文件管理

- brief 文件命名:`<YYYY-MM-DD>-<topic>.md`(YYYY-MM-DD = **数据时间戳**,不是生成时间)
- 每份 brief 头部 4 行 metadata:
  - 📌 Sample / Production 标记 + 版本号
  - 数据时间戳
  - 生成时间
  - 数据来源说明 + NO-FABRICATION 声明
